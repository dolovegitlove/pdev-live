PDev Documentation

Everything you need to know about PDev Suite and PDev Live

Understanding PDev

PDev has three parts. Here's what each one does:

Term What It Is When You Use It
PDev Suite A methodology for product development - 9 documents that take you from idea to implementation Always. It's the framework everything else is built on.
Pipeline Commands
/pdev, /idea, /spec, etc.		 Claude Code slash commands that generate pipeline documents In Claude sessions. Type these to create docs.
PDev Live This web dashboard - view, share, and manage your pipeline documents In your browser. Docs stream here automatically.

In short

Type /pdev start myproject in Claude. Documents appear here automatically. That's it.

Quick Start

1

Start a Project

/pdev start my-project

Opens the guided wizard in Claude. Streaming to PDev Live starts automatically.

2

Generate Documents

/idea /bmrk /spec

Run pipeline commands to generate each document in sequence.

3

View in PDev Live

Documents appear here in real-time as Claude generates them. No manual sync needed.

Pipeline Commands

The PDev pipeline has 9 phases. Run them in order - each builds on the previous.

/idea -> /bmrk -> /gapa -> /innov -> /caps -> /spec -> /bmrk design -> /sop -> /pv
1 /idea

Ideation

Define your project vision, target audience, core features, and success metrics.

Output: IDEATION.md
2 /bmrk

Benchmark

Competitive analysis of similar products, their strengths, weaknesses, and market positioning.

Output: BENCHMARK.md
3 /gapa

Gap Analysis

Identify gaps in the market and opportunities your product can address.

Output: GAP_ANALYSIS.md
4 /innov

Innovation

First-principles analysis of how to build something 10x better than competitors.

Output: INNOVATION.md
5 /caps

Capabilities

Technical feasibility assessment - what can be built with available resources and technology.

Output: CAPABILITIES.md
6 /spec

Product Spec

Detailed technical specification including architecture, APIs, data models, and integrations.

Output: PRODUCT_SPEC.md
7 /bmrk design

Design System

Visual and UX benchmark - colors, typography, component patterns, and interaction design.

Output: DESIGN_SYSTEM.md
8 /sop

Development SOP

Step-by-step implementation plan with phases, tasks, and deliverables.

Output: DEVELOPMENT_SOP.md
9 /pv

Pipeline Validation

Validate all pipeline documents exist and are consistent with each other.

Output: Validation Report

Orchestration Commands

These commands help you manage the pipeline without running each phase manually.

/pdev start [project]

Recommended starting point. Interactive wizard that guides you through project setup, asks the right questions, and runs pipeline commands in order. Streaming to PDev Live starts automatically.

/pdev next [project]

Automatically runs the next command in the pipeline sequence based on which documents already exist. Useful for resuming work.

/pdev status [project]

Shows pipeline progress - which documents exist, which are missing, and overall health scores.

/eval [project]

Alternative to /idea for existing products. Evaluates what you already have instead of starting from scratch.

Manual Streaming Control

Most users never need this - /pdev start handles streaming automatically. These commands give you fine-grained control when needed.

Command Purpose Example
/pdev-live start Start a streaming session manually /pdev-live start ideate MyProject
/pdev-live phase Mark a phase transition in the stream /pdev-live phase 1 Discovery
/pdev-live step Add custom content to the stream /pdev-live step "## Key Findings"
/pdev-live end End the streaming session /pdev-live end completed
/pdev-live status Check if a session is active /pdev-live status

When Would I Use This?

Manual streaming is useful when you want to:

  • Add commentary or notes between pipeline phases
  • Stream a custom workflow that doesn't follow the standard pipeline
  • Debug streaming issues

Scope Levels

PDev commands can zoom in or out depending on what you're working on.

Scope Command Example Use Case
Industry /pdev industry "wellness tech" Macro market analysis before picking a specific product idea
Product /pdev product myapp Full pipeline for a complete product (most common)
Feature /pdev feature "booking" Deep-dive on a single feature before building it
Component /pdev component "calendar" Quick benchmark before building a UI component
Sprint /pdev sprint myapp phase-2 Focus on a specific set of SOP tasks
Task /pdev task 3.2 Run mini-pipeline for a specific SOP task number

CLI Tool (Advanced)

For terminal use outside Claude sessions. Most users will never need this.

~/.claude/tools/pdev-live/client.sh start <project> <command>

Start a streaming session from your terminal. Useful for scripting or automation.

pdev-live seed [docs_path]

Manually sync existing pipeline documents to PDev Live. Use this if you edited docs outside a Claude session and want them to appear in the dashboard.

Slash Commands vs CLI Tool

Slash commands (/pdev, /pdev-live) work inside Claude sessions.

CLI tool (pdev-live, client.sh) works in your terminal outside Claude.

They do the same thing - use whichever matches where you're working.

Frequently Asked Questions

Do I have to run commands in order?

Yes, generally. Each phase builds on the previous one. However, you can skip phases if you already have that information from another source. Use /pdev status to see which documents exist.

How do I resume a project?

Just use the same project name. Running /pdev start myproject will create the project if it doesn't exist, or resume it if it does. PDev Live matches by project name automatically. Tip: use consistent naming (always "myapp", not sometimes "my-app").

Can I regenerate a document?

Yes. Running the same command again will create a new version. PDev Live tracks versions so you can see the history of changes.

Where are the documents stored?

Documents are stored in your project's dev/ directory (e.g., myproject/dev/IDEATION.md). They're also synced to PDev Live for viewing and sharing.

Will PDev Live see docs I edited outside a session?

PDev Live only auto-discovers your pipeline docs when you start or resume a session. If you modify docs outside a session, they won't appear until you either:

  • Start a new session (/pdev start myproject) - triggers auto-seed
  • Manually run pdev-live seed [docs_path] during an active session
What's the difference between /bmrk and /bmrk design?

/bmrk (phase 2) analyzes competitor products and market positioning. /bmrk design (phase 7) specifically analyzes visual design, UX patterns, and creates a design system.

Can I share documents with clients?

Yes. From any project page, click "Share with Client" to generate a time-limited guest link. Guests can view documents but cannot modify or create new ones.

Can I run PDev on a remote server?

Yes. PDev Live supports multiple server origins. Configure servers in Settings then Servers. Documents from any connected server will appear in your project library.

How do I enable Git auto-commit?

Go to Settings then Git Sync and enable automatic commits. Add your repository paths to the whitelist. PDev documents will be committed and pushed automatically.

Data Flow Step-by-Step

1

Hook Detection

When you run a PDev command (like /idea), a hook automatically detects it and triggers the streaming client.

2

Server Auto-Detection

The client identifies which server it's running on. This becomes the server_origin tag so you know where documents came from.

3

Session Creation

The client creates a session on the central API:

POST [your-pdev-host]/api/sessions
Body: { server: "server-a", project: "myproject", commandType: "idea" }

The API returns a unique session ID (UUID).

4

Step Streaming

As Claude generates output, each chunk is POSTed to the session:

POST [your-pdev-host]/api/sessions/{sessionId}/steps
Body: { type: "output", content: "## Problem Statement..." }

The server stores each step in PostgreSQL, renders markdown to HTML, and broadcasts via SSE.

5

Real-Time Browser Updates

Your browser connects to an SSE (Server-Sent Events) endpoint:

EventSource("[your-pdev-host]/api/events/{sessionId}")

Steps appear in real-time as Claude generates them. No polling, no refresh needed.

Example: Running /idea on a Remote Server

  1. You SSH into remote-server-a and start a Claude session
  2. You type /idea to start ideating
  3. Hook fires automatically, detects server as "remote-server-a"
  4. Session created at central API with server_origin: "remote-server-a"
  5. Session ID cached locally at /tmp/pdev-live-session.json
  6. All output streams to the same central API
  7. You open this dashboard - session appears tagged with its origin server

Key Insight

There's no server-to-server sync. Every server talks directly to the central API. The server_origin field tracks where each session came from, but the viewing experience is identical regardless of origin.

Why This Architecture?

Benefit Explanation
Single Source of Truth All sessions in one database. No sync conflicts.
Simple Client Just curl + jq. No special dependencies on remote servers.
Real-Time Anywhere SSE works from any browser. Open this page and watch live.
Multi-Server Visibility Dashboard shows sessions from ALL your servers in one place.
Offline Resilient If the central API is down, Claude still works - streaming just pauses.
